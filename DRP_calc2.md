# Техническое задание

## Общая задача
Моя организация занимается разработкой средства резервного копирования.
Я сейчас разрабатываю сервис, который должен помочь нашим продавцам демонстрировать преимущество нашего продукта перед конкурентами путем наглядной демонстрации снижения времени, необходимого на восстановление ИТ инфраструктуры заказчика.

Для работы приложения необходимо собрать информацию о текущей ИТ инфраструктуре заказчика на уровне количества активов различных типов, объема хранимых данных, количестве и эффективности средств резервного копирования и доступности ИТ инженеров, которые будут проводить восстановление. 

Задачей приложения будет собирать у пользователя техническую информацию о типах восстанавливаемых систем (серверы, рабочие станции, базы данных, файловые хранилища), и количество доступных ресурсах для восстановления (т.е. количество ИТ инженеров). 

Используя данные об объеме восстанавливаемой информации и ресурсах, приложение будет использовать стохастические механизмы моделирования (метод Монте-Карло), чтоб получить лучших и худший сценарий восстановления, а также вероятность получения лучшего сценария. 

Информация будет выводится в виде диаграммы плотности распределения и данных о времени восстановления отдельных типов активов, выполненного в виде таймлайна. 

Подробные описания объектов восстановления, дополнительные условия и другие подробности даны ниже. Все данные даны с нужными тегами и разметкой.

## Ресурсы процесса восстановления

### Инженеры ИТ
Инженеры ИТ обеспечивают:
* работу систем восстановления
* планирование и исполнение задач по восстановлению
* оценка успешности восстановления
* дополнительная настройка перед запуском

Количество инженеров ИТ должно быть задано в процессе сбора информации.

Мы будем считать, что инженеры работают сменами по 8 часов 7 дней в неделю. Количество инженеров в расчетах должно исходить из того что процесс восстановления должен поддерживаться круглосуточно без перерывов.
### Системы хранения 
#### Ленточная библиотека
Основной параметр - количество драйвов. Значение умолчанию = 4.
Количество потоков восстановления = количество драйвов
Стандартная скорость восстановления по одному потоку = 300 Мб/Сек.  
Общая скорость восстановления ограничена скоростью сети передачи данных

#### СХД хранилище 
Базируется на RAID массиве с максимальной скоростью восстановления = 4 ГБ/сек. 
Количество потоков восстановления = 10
Стандартная скорость восстановления по одному потоку = 400 Мб/Сек
Общая скорость восстановления ограничена скоростью сети передачи данных

### Сеть передачи данных
#### Скоростная сеть
Сеть между ленточной библиотекой и СХД.
Она используется в сценариях расчета Конкурента
По умолчанию скорость 10 Гб\сек

#### Сеть ЛВС
Сеть между:
* ленточной библиотекой и Объектами восстановления
* СХД и объектами восстановления
Она используется:
* в сценарии Киберпротект используется только эта сеть
* в сценарии Конкурент сеть ЛВС используется во второй фазе восстановления
По умолчанию скорость 1 Гб\сек

## Логика расчета времени восстановления

Логика всего процесса - системы хранения передают данные на восстанавливаемые объекты восстановления, создавая пул систем, готовых к запуску. Инженеры ИТ работают с этим пулом систем, разбирая его параллельными потоками, с темпом, зависящим от параметров восстановления.

Восстановление будет идти в определенной последовательности:
* Базы данных
* Сервера
* Файловые хранилища
* Рабочие станции

Данные будут восстанавливаться с систем хранения заданным количеством потоков со скоростью, которая определена в свойствах системы хранения.

Для расчета количества доступных потоков восстановления будем брать общее количество всех драйвов всех имеющихся библиотек. Общая скорость восстановления тем не менее будет ограничена пропускной способностью сети.

Системы хранения начинают восстановление следующей "волны" объектов восстановления сразу же после того, как полностью завершена передача данных на предыдущую "волну"

Как только передача данных закончена, доступные инженеры ИТ начинают проверку и запуск восстановленных объектов восстановления. Если количество доступных инженеров больше, чем количество готовых к проверке объектов восстановления, свободные инженерны ИТ ничего не делают. 

Мы будем делать два расчета, для Кибербакап и для Конкурента. Исходные данные м общий принцип будут одинаковы. Разница в описана ниже в секции Кибербакап и Конкурент

### Кибербакап

Проводит восстановление в 10 одновременных потоков. Данные передаются непосредственно на объекты восстановления, согласно последовательности восстановления. 
Ограничения передачи данных равны параметрам скоростной сети.

Скорость восстановления будет рассчитана как суммарная скорость всех доступных драйвов всех ленточных библиотек разделенная на количество одновременных потоков.

СХД не принимает участие в процессе восстановления

### Конкурент

Восстановление вначале проходит на СХД. Скорость восстановления будет рассчитана как суммарная скорость всех доступных драйвов всех ленточных библиотек. 
Ограничения передачи данных равны параметрам скоростной сети.

Восстановление объектов восстановления проводится с СХД в 10 потоков, ограниченных скоростью ЛВС.

### Шаг 3 - Результаты

Выполняется Monte Carlo симуляция с заданным значением триалов.
Моделирование выполняется отдельно для вариантов расчета Кибербакап и Конкурент
Результаты визуализируются через две диаграммы плотности распределения.

- Рассчитать общий объем данных для каждого типа системы: количество x средний объем
- Рассчитать объем данных на одну ленточную библиотеку: общий объем всех данных / количество библиотек
- Для каждого типа системы рассчитать базовое время восстановления: объем данных в системах этого типа / (количество драйвов * пропускная способность драйва)
- Следовать разным подходам в зависимости от вариантов Кибербакап или Конкурент

- Запустить моделирование:

        - Варианты скорости сети (случайный коэффициент от 0.8 до 1.2 от номинальной пропускной способности)
    - Применить все факторы к времени восстановления (базовое время + время настройки) для каждого типа

- Собрать массив итоговых времен восстановления (одно значение на триал, всего 1000 значений)

## Ключевые решения при планировании — Фаза 1

### Модель инженеров ИТ

- **Количество верификаторов на смену:** `floor(N ÷ 3)`, где N — количество инженеров, введённых пользователем. При N = 3 → 1 верификатор на смену, при N = 6 → 2 и т.д.
- **Если N < 3:** часть смен будет без верификаторов — к каждой такой смене добавляется 480 минут (8-часовая пауза на один актив). Минимально допустимый ввод = 1 (0 запрещён формой).
- **Верификация последовательная:** 1 инженер обрабатывает 1 объект одновременно.
- **Инженер начинает сразу:** как только завершена передача данных на конкретный объект — не ждёт окончания всей "волны". Передача следующей волны идёт параллельно с верификацией текущей.
- **Случайный фактор инженерной доступности:** `engineerFactor` = равномерное распределение `Uniform(0.5, 1.0)` — отражает снижение реальной доступности.

### Модель восстановления и запуска активов

- **Порядок восстановления** фиксированный: БД → Серверы → Файловые хранилища → Рабочие станции (`ASSET_PRIORITY`). Не конфигурируется.
- **Время запуска после восстановления:** фиксированные **30 минут** для всех типов активов (БД, Сервер, ФС, РС) — константа `STARTUP_MINUTES`, без стохастики.
  - *Примечание:* ТЗ описывало разное время для разного оборудования (15 мин / 1–3 ч в зависимости от того, то же оборудование или другое) — это упрощено до единой константы на этапе планирования. Усложнение отложено.
- **Поправка на неопределённость (`uncertaintyFactor`)** применяется **только к времени передачи данных**, но никогда — к `STARTUP_MINUTES`. Значение: `1 + Uniform(0, uncertaintyPct)`, один розыгрыш на актив на триал.
- **Фактор сети (`networkFactor`):** `1 + Uniform(-0.2, 0.2)`, ограничен диапазоном `[0.8, 1.2]`. Один розыгрыш на триал, **общий для обоих сценариев** (Кибербакап и Конкурент) — это гарантирует честное сравнение: разница в результатах обусловлена логикой, а не случайным шумом.

### Модель симуляции

- **Оба сценария** (Кибербакап и Конкурент) вычисляются в одном цикле: одни и те же случайные розыгрыши (`networkFactor`, `engineerFactor`) на каждый триал.
- **Единицы:** внутри движка всё считается в **минутах**, конвертация в **часы** происходит однократно на выходе — перед построением `ScenarioResult` и KDE.
- **Распределение данных между библиотеками:** `totalData / количество_библиотек` — данные делятся поровну.
- **Кибербакап** — прямая передача лента → объекты, 10 потоков, скорость ограничена быстрой сетью.
- **Конкурент** — двухфазная передача: фаза 1 лента → СХД (быстрая сеть), фаза 2 СХД → объекты (ЛВС).
